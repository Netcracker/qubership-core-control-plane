# Cloud Core Mesh CRs → Istio resources

## Overview

This skill converts Core mesh CRs to Istio resources:

| Source CR | apiVersion | Output |
|---|---|---|
| `StatefulSession` | `nc.core.mesh/v3` | `DestinationRule` with `consistentHash.httpCookie` |
| `LoadBalance` | `nc.core.mesh/v2` or `nc.core.mesh/v3` | `DestinationRule` with `consistentHash.*` |
| `HttpFilters` + `RouteConfiguration` (rules with `luaFilter`) | `core.netcracker.com/v1` `Mesh` (`subKind`) or legacy `nc.core.mesh/v3` | `EnvoyFilter` (&lt; 1.30) or `TrafficExtension` (≥ 1.30) |

> **Out of scope:** `StatefulSession` embedded inside a `RouteConfiguration` rule
> is handled by the [`core-mesh-crs-to-gatewayapi`](../core-mesh-crs-to-gatewayapi/SKILL.md) skill
> as part of HTTPRoute generation.

The transformation keeps charts deployable to **both mesh types** by wrapping originals in
`{{- if eq .Values.SERVICE_MESH_TYPE "Core" }}` and generated resources in
`{{- if eq .Values.SERVICE_MESH_TYPE "Istio" }}`.

---

## Scope

Files containing any of:

- standalone `StatefulSession` or `LoadBalance` CR documents
- `Mesh` CRs with `subKind: HttpFilters` and/or `subKind: RouteConfiguration` (rules with `luaFilter`)
- legacy `HttpFilters` / `RouteConfiguration` documents (`nc.core.mesh/v3`) in declarative config files

Do not touch Deployments, Services, helpers (`*.tpl`), or unrelated CR kinds.
`HttpFilters` entries for `wasmFilters` / `extAuthzFilter` alone (no `luaFilters`) are out of scope.

---

## Step-by-Step Procedure

### Step 1 — Discover

```bash
grep -rl \
  -e 'kind: StatefulSession' \
  -e 'kind: LoadBalance' \
  -e 'subKind: HttpFilters' \
  -e 'kind: HttpFilters' \
  -e 'luaFilter:' \
  --include="*.yaml" --include="*.yml" \
  <folder>
```

List each discovered file and its contained kinds before proceeding.

**Scope gate:** treat as in scope when the document is either:
- `apiVersion: core.netcracker.com/v1`, `kind: Mesh`, `subKind: HttpFilters` or `RouteConfiguration`, or
- legacy `apiVersion: nc.core.mesh/v2` / `nc.core.mesh/v3` with `kind: HttpFilters` or `RouteConfiguration`.

Drop false positives (e.g. Istio `EnvoyFilter` output files).

For Lua migration, skill input is **one pair** (`HttpFilters` + `RouteConfiguration` with
matching gateways and `luaFilter` references). When a chart defines Lua on several gateways,
run the transformation once per pair.

Resolve Istio version (`.Values.ISTIO_VERSION` or ask user) before choosing Lua output — see
[lua-filter-mapping.md](lua-filter-mapping.md). Dataplane is **ambient only**.

### Step 2 — Wrap originals in Core condition

Wrap each Core mesh document in the original file:

```yaml
{{- if eq .Values.SERVICE_MESH_TYPE "Core" }}
apiVersion: core.netcracker.com/v1
kind: Mesh
subKind: HttpFilters
# ... original spec unchanged ...
{{- end }}
```

Legacy declarative files keep `apiVersion: nc.core.mesh/v3` inside the Core guard.

For multi-document YAML files: wrap each document individually.

### Step 3 — Generate Istio files

Create a new `-istio` sibling file for each original:

```
templates/stateful-session.yaml  → templates/stateful-session-istio.yaml
templates/load-balance.yaml      → templates/load-balance-istio.yaml
templates/lua-filter.yaml        → templates/lua-filter-istio.yaml
```

Each autogenerated file must be wrapped in the ISTIO guard:

```yaml
{{- if eq .Values.SERVICE_MESH_TYPE "Istio" }}
# ... DestinationRule resources ...
{{- end }}
```

**StatefulSession → DestinationRule:** follow [stateful-session-mapping.md](stateful-session-mapping.md).  
**LoadBalance → DestinationRule:** follow [load-balance-mapping.md](load-balance-mapping.md).  
**HttpFilters + RouteConfiguration (luaFilter) → EnvoyFilter / TrafficExtension:** follow [lua-filter-mapping.md](lua-filter-mapping.md).

### Step 4 — Validation checklist

- [ ] Every original file's CRs are wrapped in `Core` condition
- [ ] Every generated file's resources are wrapped in `Istio` condition
- [ ] `StatefulSession` with cookie → `DestinationRule` generated; delete/disabled requests skipped
- [ ] `StatefulSession` with `hostname`/`port` → `# ⚠ MANUAL REVIEW` comment added (endpoint-level targeting)
- [ ] `LoadBalance` → `DestinationRule` generated from first policy; multi-policy cases get `# ⚠ MANUAL REVIEW` comment
- [ ] `HttpFilters` + `RouteConfiguration` with `luaFilter` → `EnvoyFilter` (&lt; 1.30) or `TrafficExtension` (≥ 1.30) per Istio version
- [ ] Istio &lt; 1.30: ingress/egress → one `EnvoyFilter` with `context: GATEWAY`, base filter + `LuaPerRoute` (multiple gateways may share one CR via `targetRefs`); waypoint → **separate** `EnvoyFilter`, `HTTP_FILTER` without `context`, full `inline_code` + path guard
- [ ] Istio ≥ 1.30: all gateways → `TrafficExtension` with `targetRefs`, `phase: STATS`, path guard in `inlineCode`
- [ ] Core `RouteConfiguration` on public/private/internal gateway → `virtualServices[].name` matches gateway name, `hosts: ["*"]`
- [ ] Unresolved `luaFilter` names, route names, or gateway context → `# ⚠ MANUAL REVIEW` comment added
- [ ] `overridden: true` on any CR → `# ⚠ MANUAL REVIEW` comment added
- [ ] No hardcoded values where Helm expressions existed
- [ ] YAML is valid

### Fields that MUST be flagged with `# ⚠ MANUAL REVIEW`

| Source | Field | Trigger |
|---|---|---|
| `StatefulSession.spec` | `hostname` | non-empty |
| `StatefulSession.spec` | `port` | non-empty |
| `StatefulSession.spec` | `overridden` | non-empty |
| `LoadBalance.spec` | `overridden` | non-empty |
| `LoadBalance.spec.policies` | more than one entry | — |
| `Rule` | `luaFilter` | name not found in `HttpFilters.spec.luaFilters` |
| `EnvoyFilter` patch | `route.name` | cannot be resolved from generated HTTPRoute (ingress/egress only) |
| Waypoint `EnvoyFilter` | path-scoped script | same `luaFilter` name used on rules with different prefixes |
| `HttpFilters` / `RouteConfiguration` | `gateways` | gateway context cannot be classified |
